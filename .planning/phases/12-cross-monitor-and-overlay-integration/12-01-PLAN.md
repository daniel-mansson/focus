  <action>
Add a new public static method `FindAdjacentMonitor` to MonitorHelper that finds the adjacent monitor in a given direction using the overlapping-range adjacency algorithm.

Signature:
```csharp
public static unsafe (RECT Work, RECT Monitor)? FindAdjacentMonitor(
    HMONITOR current, RECT currentMonitorRect, string direction)
```

Algorithm:
1. Enumerate all monitors via `EnumDisplayMonitors` (reuse existing delegate pattern with GC-safe local variable)
2. For each candidate monitor (skip if same as `current`), get its `MONITORINFO` for `rcMonitor` and `rcWork`
3. Test adjacency based on direction:
   - "right": `Math.Abs(candidate.rcMonitor.left - currentMonitorRect.right) <= 2` (2px edge tolerance for non-flush arrangements) AND candidate overlaps vertically with current
   - "left": `Math.Abs(candidate.rcMonitor.right - currentMonitorRect.left) <= 2` AND vertical overlap
   - "down": `Math.Abs(candidate.rcMonitor.top - currentMonitorRect.bottom) <= 2` AND horizontal overlap
   - "up": `Math.Abs(candidate.rcMonitor.bottom - currentMonitorRect.top) <= 2` AND horizontal overlap
4. Perpendicular overlap = `Math.Max(0, Math.Min(a.bottom, b.bottom) - Math.Max(a.top, b.top))` for horizontal direction, or analogous for vertical
5. If multiple candidates, pick the one with the most perpendicular overlap
6. Return `(rcWork, rcMonitor)` tuple or null if no adjacent monitor found

Use `Windows.Win32.Graphics.Gdi.MONITORINFO` and `MONITORENUMPROC` via CsWin32. Ensure the `MONITORENUMPROC` callback delegate is stored in a local variable to prevent GC collection during enumeration (same pattern as existing `EnumerateMonitors`).

Add necessary using directives: `using Windows.Win32.Graphics.Gdi;` is already in scope. Ensure `HMONITOR` comparison uses `(nint)(IntPtr)` casting for handle equality, matching existing code patterns.

Extend `MoveOrResize` to detect when a Move operation hits the monitor boundary and transition the window to the adjacent monitor. This only applies to `WindowMode.Move`, NOT `WindowMode.Grow`.

**Step 1: Add a private helper `TryGetCrossMonitorTarget`:**

```csharp
private static unsafe (RECT work, RECT monitor)? TryGetCrossMonitorTarget(
    HWND hwnd, string direction, RECT vis, RECT currentWork)
```

Logic:
1. Get the HMONITOR for `hwnd` via `MonitorFromWindow(hwnd, MONITOR_DEFAULTTONEAREST)`
2. Get its `MONITORINFO` to obtain `rcMonitor`
3. Check if the window's visible bounds are at the work area boundary:
   - "right": `vis.right >= currentWork.right`
   - "left": `vis.left <= currentWork.left`
   - "down": `vis.bottom >= currentWork.bottom`
   - "up": `vis.top <= currentWork.top`
4. If not at boundary, return null
5. Call `MonitorHelper.FindAdjacentMonitor(hMon, mi.rcMonitor, direction)` and return result

**Step 2: Add a private helper `ComputeCrossMonitorPosition`:**

```csharp
private static RECT ComputeCrossMonitorPosition(
    string direction, RECT vis, RECT win, RECT targetWork,
    int tStepX, int tStepY,
    int borderL, int borderT, int borderR, int borderB)
```

Logic per locked decisions from CONTEXT.md:
- Movement axis snaps to first grid cell from the target monitor edge
- Perpendicular axis preserves current pixel position (clamped to target work area)
- If window is larger than target work area, clamp size (do not resize)

For direction "right": `newVisLeft = targetWork.left + tStepX`, perpendicular `newVisTop = Math.Clamp(vis.top, targetWork.top, targetWork.bottom - clampedH)`
For direction "left": `newVisLeft = targetWork.right - tStepX - clampedW`, perpendicular same
For direction "down": `newVisTop = targetWork.top + tStepY`, perpendicular `newVisLeft = Math.Clamp(vis.left, targetWork.left, targetWork.right - clampedW)`
For direction "up": `newVisTop = targetWork.bottom - tStepY - clampedH`, perpendicular same

Where `clampedW = Math.Min(visW, targetWork.right - targetWork.left)` and `clampedH = Math.Min(visH, targetWork.bottom - targetWork.top)`.

Return the new RECT in GetWindowRect coordinate space (apply border offsets).

**Step 3: Integrate into MoveOrResize:**

After the existing `RECT newWinRect = mode switch { ... }` computation, add a cross-monitor check for Move mode ONLY:

```csharp
if (mode == WindowMode.Move)
{
    // Check if the computed position is at the boundary (cross-monitor candidate)
    // Use the NEW visible position after ComputeMove, not the original
    int newVisLeft = newWinRect.left + borderLeft;
    int newVisTop  = newWinRect.top  + borderTop;
    int newVisRight = newWinRect.right - borderRight;
    int newVisBottom = newWinRect.bottom - borderBottom;

    bool atBoundary = direction switch
    {
        "right" => newVisLeft + (newVisRight - newVisLeft) >= workArea.right,
        "left"  => newVisLeft <= workArea.left,
        "down"  => newVisTop + (newVisBottom - newVisTop) >= workArea.bottom,
        "up"    => newVisTop <= workArea.top,
        _       => false
    };

    if (atBoundary)
    {
        var target = TryGetCrossMonitorTarget(fgHwnd, direction, visRect, workArea);
        if (target.HasValue)
        {
            var (targetWork, _) = target.Value;
            int tw = targetWork.right - targetWork.left;
            int th = targetWork.bottom - targetWork.top;
            var (tStepX, tStepY) = GridCalculator.GetGridStep(tw, th, config.GridFractionX, config.GridFractionY);

            newWinRect = ComputeCrossMonitorPosition(
                direction, visRect, win, targetWork, tStepX, tStepY,
                borderLeft, borderTop, borderRight, borderBottom);
        }
    }
}
```

IMPORTANT: The cross-monitor check must happen AFTER ComputeMove computes the clamped position. If the clamped result is at the boundary, we know the window was trying to move past the edge. Then we check for an adjacent monitor and jump.

Also rename local variables in MoveOrResize to avoid conflicts: the existing `winRect` should stay as `winRect` (it's used in ComputeMove), add alias `win` for passing to ComputeCrossMonitorPosition.

CRITICAL ANTI-PATTERN: Do NOT mix `rcMonitor` and `rcWork` coordinates. Adjacency detection uses `rcMonitor` edges (physical screen edges). Window placement uses `rcWork` (excludes taskbar). The `TryGetCrossMonitorTarget` boundary check uses `currentWork` (rcWork).

Add `using Focus.Windows;` if not already present (for `MonitorHelper` access). `MonitorHelper` is in namespace `Focus.Windows`, and `WindowManagerService` is in `Focus.Windows.Daemon` which should already have access.
