  <action>
In `OnDirectionKeyDown`, change the non-Navigate branch from a single-expression lambda to a block lambda that refreshes ONLY the active window overlay after `MoveOrResize`, guarded by `_capsLockHeld`.

**Key design decision:** Do NOT call `ShowOverlaysForCurrentForeground()` — that would also draw navigate-target outlines for surrounding windows. In Move/Grow/Shrink modes, only the active window outline should be visible. Instead, use `_overlayManager.HideAll()` + `_overlayManager.ShowForegroundOverlay()` to refresh just the active border.

Change:
```csharp
if (mode != WindowMode.Navigate)
{
    try
    {
        _staDispatcher.Invoke(() => WindowManagerService.MoveOrResize(direction, mode, _verbose));
    }
    catch (ObjectDisposedException) { }
    catch (InvalidOperationException) { }
    return;
}
```

To:
```csharp
if (mode != WindowMode.Navigate)
{
    try
    {
        _staDispatcher.Invoke(() =>
        {
            WindowManagerService.MoveOrResize(direction, mode, _verbose);
            if (_capsLockHeld)
                RefreshForegroundOverlayOnly();
        });
    }
    catch (ObjectDisposedException) { }
    catch (InvalidOperationException) { }
    return;
}
```

Add a new private method `RefreshForegroundOverlayOnly()` that:
1. Calls `_overlayManager.HideAll()` to clear all overlays (including stale navigate targets)
2. Gets the foreground window via `PInvoke.GetForegroundWindow()`
3. Gets its visible bounds via `DwmGetWindowAttribute(DWMWA_EXTENDED_FRAME_BOUNDS)`
4. Calls `_overlayManager.ShowForegroundOverlay(bounds, ForegroundBorderColor)` to show only the active window outline

This method should follow the same guard pattern as ShowOverlaysForCurrentForeground (return silently if GetForegroundWindow returns default or DwmGetWindowAttribute fails).

Why the `_capsLockHeld` guard: If the user releases CapsLock between the direction keydown event and the STA dispatch executing, the overlays should not be shown (they were already hidden by OnReleasedSta). Without the guard, a race condition could flash overlays briefly after release.

Why RefreshForegroundOverlayOnly instead of ShowOverlaysForCurrentForeground: During Move/Grow/Shrink the user is manipulating the active window — showing navigate targets is distracting and wrong. Only the active window border should track the window's new position.

Do NOT add any timer or delay — the overlay refresh should be synchronous and immediate within the STA dispatch, matching the feel of instant grid snapping.
