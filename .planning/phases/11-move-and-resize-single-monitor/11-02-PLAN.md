**WindowManagerService.cs — Fix ComputeShrink edge mapping (swap all four cases):**

The current code moves the edge NAMED by the direction (e.g., "up" moves top edge). The user expects the OPPOSITE edge to move (e.g., "up" should move the bottom edge upward). Rewrite the switch body:

- case "up": Move BOTTOM edge upward. `newVisBottom` changes, `newVisTop` stays fixed.
  ```csharp
  case "up":
      if (visH <= stepY) return win;
      newVisBottom = GridCalculator.IsAligned(vis.bottom, work.top, stepY, tolY)
          ? vis.bottom - stepY
          : GridCalculator.NearestGridLineFloor(vis.bottom, work.top, stepY);
      newVisBottom = Math.Max(newVisBottom, newVisTop + stepY);
      break;
  ```

- case "down": Move TOP edge downward. `newVisTop` changes, `newVisBottom` stays fixed.
  ```csharp
  case "down":
      if (visH <= stepY) return win;
      newVisTop = GridCalculator.IsAligned(vis.top, work.top, stepY, tolY)
          ? vis.top + stepY
          : GridCalculator.NearestGridLineCeiling(vis.top, work.top, stepY);
      newVisTop = Math.Min(newVisTop, newVisBottom - stepY);
      break;
  ```

- case "left": Move RIGHT edge leftward. `newVisRight` changes, `newVisLeft` stays fixed.
  ```csharp
  case "left":
      if (visW <= stepX) return win;
      newVisRight = GridCalculator.IsAligned(vis.right, work.left, stepX, tolX)
          ? vis.right - stepX
          : GridCalculator.NearestGridLineFloor(vis.right, work.left, stepX);
      newVisRight = Math.Max(newVisRight, newVisLeft + stepX);
      break;
  ```

- case "right": Move LEFT edge rightward. `newVisLeft` changes, `newVisRight` stays fixed.
  ```csharp
  case "right":
      if (visW <= stepX) return win;
      newVisLeft = GridCalculator.IsAligned(vis.left, work.left, stepX, tolX)
          ? vis.left + stepX
          : GridCalculator.NearestGridLineCeiling(vis.left, work.left, stepX);
      newVisLeft = Math.Min(newVisLeft, newVisRight - stepX);
      break;
  ```

Note: ComputeShrink also uses the new directional snap variants (Floor for edges moving inward toward smaller values, Ceiling for edges moving inward toward larger values) to avoid the same wrong-direction snap bug that affected ComputeGrow.

**WindowManagerService.cs — Add post-computation no-op guard for OS minimum size:**

After the switch block in ComputeShrink (after all four cases), before the "translate back to winRect space" return, add:

```csharp
// Post-computation no-op guard: if the visible dimension did not actually shrink,
// SetWindowPos would only move the window (OS clamps to ptMinTrackSize). Return unchanged.
int newVisW = newVisRight - newVisLeft;
int newVisH = newVisBottom - newVisTop;
if (newVisW >= visW && newVisH >= visH) return win;
```

This catches the case where the window is at OS minimum size (larger than one grid step) and the grid math produces a rect that would require SetWindowPos to shrink below OS minimum. Without this guard, SetWindowPos would honor the new position but clamp the size, causing the window to move without shrinking.
