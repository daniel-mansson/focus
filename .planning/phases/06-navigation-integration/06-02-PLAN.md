**Part A: Add ForceCapsLockOff static method**

Add a `private static void ForceCapsLockOff()` method to `DaemonCommand`. It checks if CAPSLOCK toggle is ON (low-order bit of `PInvoke.GetKeyState(0x14)`) and if so, sends a synthetic VK_CAPITAL down+up via `PInvoke.keybd_event`:
```csharp
private static void ForceCapsLockOff()
{
    const int VK_CAPITAL = 0x14;
    if ((PInvoke.GetKeyState(VK_CAPITAL) & 0x0001) != 0)
    {
        PInvoke.keybd_event((byte)VK_CAPITAL, 0x45, 0, 0);
        PInvoke.keybd_event((byte)VK_CAPITAL, 0x45, KEYBD_EVENT_FLAGS.KEYEVENTF_KEYUP, 0);
    }
}
```
Note: Check the CsWin32-generated signature for `keybd_event` — the flags parameter may be `KEYBD_EVENT_FLAGS` enum or `uint`. Adjust the constant accordingly. `KEYEVENTF_KEYUP = 0x0002`.

**Part B: Modify DaemonCommand.Run to create OverlayOrchestrator**

The challenge: OverlayOrchestrator must be created ON the STA thread (it creates a Control and installs WinEvent hook). But DaemonCommand.Run creates the STA thread. Solution: Create OverlayOrchestrator inside DaemonApplicationContext constructor (which runs on the STA thread).

Changes to DaemonCommand.Run:
1. Before creating components, call `ForceCapsLockOff();` (must be before hook install — keyboard hook would eat it)
2. Load config: `var config = FocusConfig.Load();`
3. Change DaemonApplicationContext constructor to accept `FocusConfig config` and `bool verbose` in addition to existing params. The ApplicationContext will create OverlayOrchestrator on the STA thread and inject callbacks into the monitor.
4. The problem: CapsLockMonitor needs the callbacks at construction time, but the callbacks come from OverlayOrchestrator which is created on the STA thread later. Solution: Use a "late-binding" approach — create CapsLockMonitor with `Action` closures that capture a mutable `OverlayOrchestrator?` reference, then set that reference once the STA thread creates the orchestrator.

Concrete implementation:
```csharp
// In DaemonCommand.Run:
ForceCapsLockOff();
var config = FocusConfig.Load();

// ... channel creation, hook creation as before ...

// Late-binding reference — set by DaemonApplicationContext on STA thread
OverlayOrchestrator? orchestrator = null;

var monitor = new CapsLockMonitor(channel.Reader, verbose,
    onHeld: () => orchestrator?.OnCapsLockHeld(),
    onReleased: () => orchestrator?.OnCapsLockReleased());

// ... consumer task, staThread as before, but:
var staThread = new Thread(() =>
{
    Application.Run(new DaemonApplicationContext(hook, monitor, () => cts.Cancel(), config, verbose, out orchestrator));
});
```

5. In the cleanup section (after cancellation), before hook.Uninstall:
   - Call `orchestrator?.RequestShutdown();` — prevents Invoke exceptions during teardown.
   - After staThread.Join, call `orchestrator?.Dispose();` — disposes ForegroundMonitor, timers, Control (all on main thread — STA thread has exited by now, but Dispose just releases resources).

**Part D: Add fade animation to OverlayOrchestrator**

Modify OverlayOrchestrator.cs to add fade-in and fade-out:

1. Add `_fadeTimer.Tick += OnFadeTick;` in constructor.

2. Modify `OnHeldSta()`: After calling `ShowOverlaysForCurrentForeground()`, call `StartFadeIn()`.

3. Modify `OnReleasedSta()`: Instead of calling `_overlayManager.HideAll()` immediately, call `StartFadeOut()`. The overlays will be hidden after the fade-out completes.

4. Add `private void StartFadeIn()`:
   - Stop any running fade timer.
   - Set `_fadingIn = true;`
   - Set `_fadeProgress = 0.0f;` (start from transparent — or from current progress if re-entering)
   - Repaint all overlays at current progress (so first frame appears immediately).
   - Start `_fadeTimer`.

5. Add `private void StartFadeOut()`:
   - Stop any running fade timer (handles case where fade-in was mid-progress — Pitfall 7).
   - Set `_fadingIn = false;`
   - `_fadeProgress` stays at wherever the fade-in left off (smooth transition).
   - Start `_fadeTimer`.

6. Add `private void OnFadeTick(object? sender, EventArgs e)`:
   - Calculate step: `float step = _fadingIn ? (16f / 100f) : (16f / 80f);` (100ms fade-in, 80ms fade-out).
   - Update progress: `_fadeProgress = _fadingIn ? Math.Min(1.0f, _fadeProgress + step) : Math.Max(0.0f, _fadeProgress - step);`
   - Call `RepaintAllOverlays(_fadeProgress);`
   - Check completion: `bool done = _fadingIn ? _fadeProgress >= 1.0f : _fadeProgress <= 0.0f;`
   - If done: stop `_fadeTimer`. If fade-out completed: call `_overlayManager.HideAll();`

7. Add `private void RepaintAllOverlays(float fadeProgress)`:
   - This needs to re-call `_renderer.Paint` on each visible overlay with alpha scaled by fadeProgress.
   - Problem: OverlayManager.ShowOverlay calls renderer.Paint internally with the configured color. We need a way to repaint with scaled alpha.
   - Solution: Add a `RepaintOverlays(float alphaScale)` method to OverlayManager that re-paints all currently shown overlays with their last-known bounds and colors, but with alpha multiplied by alphaScale. OR: Track the last-shown bounds and colors per direction in OverlayOrchestrator and call `ShowOverlay` again with alpha-scaled colors.
   - **Recommended approach:** Store `_lastBounds` (Dictionary<Direction, RECT>) and `_lastColors` (Dictionary<Direction, uint>) in OverlayOrchestrator. When `ShowOverlaysForCurrentForeground` is called, populate these. In `RepaintAllOverlays(float fadeProgress)`, iterate over stored bounds/colors, scale the alpha component of each color: `uint scaledColor = (color & 0x00FFFFFF) | ((uint)(((color >> 24) & 0xFF) * fadeProgress) << 24);` then call `_overlayManager.ShowOverlay(dir, bounds, scaledColor)`.

8. Modify `OnForegroundChanged`: If CAPSLOCK is held AND a fade-in is in progress or complete, cancel any in-progress fade, set progress to 1.0 (instant), call ShowOverlaysForCurrentForeground, then immediately repaint at current full opacity. If fade-out is running (CAPSLOCK released), ignore foreground changes.

9. Modify `OnReleasedSta` to also clear `_lastBounds` and `_lastColors` after fade-out completes (in OnFadeTick done handler).

**Important:** The fade feature means `OnReleasedSta` does NOT immediately hide overlays. Instead:
- It starts fade-out. Overlays become gradually transparent over ~80ms.
- When fade-out completes (progress <= 0), `_overlayManager.HideAll()` is called.
- This matches the user decision: "Quick fade-out (~80ms) when CAPSLOCK is released."
