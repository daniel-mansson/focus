### Config Consistency (NAV-02)

Daemon and CLI use identical call sequences:
1. `FocusConfig.Load()` → `WindowEnumerator.GetNavigableWindows()` → `ExcludeFilter.Apply()` → `NavigationService.GetRankedCandidates()` → `FocusActivator.ActivateWithWrap()`
2. No modifications to any shared service class. All are stateless static classes.

### Design Note

Dual-config pattern: `_config` (startup) for overlay display vs `FocusConfig.Load()` (fresh per keypress) for navigation. By design per Phase 8 decision. Phase 9 may need to reconsider live config reload for overlay-refresh behavior.

## E2E Flows

### Flow 1: CAPSLOCK + Direction Key Navigation

User holds CAPSLOCK → overlay shows → user presses direction key → key suppressed from app → direction event flows through channel → CapsLockMonitor dispatches → callback fires → OverlayOrchestrator marshals to STA → NavigateSta runs full pipeline → focus switches to target window.

**Status:** COMPLETE (12/12 steps verified, human-tested)

### Flow 2: CLI Independence

`Program.cs` has zero imports from daemon namespace. Shared services (`NavigationService`, `FocusActivator`, etc.) are stateless and used identically by both paths. CLI works correctly with daemon stopped.

**Status:** COMPLETE (human-verified)

### Flow 3: Overlay Chaining (Phase 9)

User holds CAPSLOCK → navigates → overlay persists → navigates again → overlay refreshes to new candidates → repeat.

**Status:** NOT IMPLEMENTED (Phase 9 not started)
