Modify HookCallback in KeyboardHookHandler to intercept 8 direction keys when CAPSLOCK is held:

**Add virtual key code constants:**
```
VK_LEFT = 0x25, VK_UP = 0x26, VK_RIGHT = 0x27, VK_DOWN = 0x28
VK_W = 0x57, VK_A = 0x41, VK_S = 0x53, VK_D = 0x44
```

**Add CAPSLOCK state tracking:**
- Add a `private bool _capsLockHeld;` instance field
- In the existing CAPSLOCK handling block (where `kbd->vkCode == VK_CAPITAL`), update `_capsLockHeld` based on `isKeyDown` BEFORE the TryWrite and suppress-return. This gives real-time state for the direction key check.

**Add direction key interception logic BEFORE the existing CAPSLOCK check:**
After the injected-event filter (LLKHF_INJECTED check) and before the `vkCode != VK_CAPITAL` early-return, add a direction key handler block:

```
1. Check if kbd->vkCode is one of the 8 direction VK codes (use a static HashSet<uint> or a switch expression for O(1) lookup)
2. If it IS a direction key:
   a. If _capsLockHeld is false → CallNextHookEx (pass through normally — HOTKEY-04)
   b. If _capsLockHeld is true:
      - Determine if this is keydown: isKeyDown = (wParam == WM_KEYDOWN || wParam == WM_SYSKEYDOWN)
      - Read modifier state: shift = (GetKeyState(VK_SHIFT) & 0x8000) != 0, ctrl = (GetKeyState(VK_CONTROL) & 0x8000) != 0, alt = (kbd->flags & LLKHF_ALTDOWN) != 0
      - TryWrite a new KeyEvent(kbd->vkCode, isKeyDown, kbd->time, shift, ctrl, alt) to the channel
      - Return (LRESULT)1 to suppress the key from reaching the focused app (HOTKEY-03)
3. If it is NOT a direction key → fall through to existing CAPSLOCK check
```

**Key behavior per user decisions:**
- CAPSLOCK + any modifier (Shift/Ctrl/Alt/Win) + direction = still suppressed (CAPSLOCK held means daemon owns direction keys regardless of other modifiers). Do NOT add modifier checks for direction keys.
- Only the 8 keys listed — no numpad arrows (numpad uses different VK codes: VK_NUMPAD2/4/6/8)
- CAPSLOCK must already be held when direction key is pressed — the `_capsLockHeld` check handles this naturally since it's set on CAPSLOCK keydown
- Both keydown and keyup for direction keys are posted to the channel (keyup needed for Phase 8 repeat prevention — per CONTEXT "Claude's Discretion: Whether to track keyup events for direction keys")

**Important: Do NOT change the CAPSLOCK handling behavior** — CAPSLOCK itself is still fully suppressed (return (LRESULT)1) regardless of held state. Only direction keys have the conditional pass-through logic.

**WASD case handling (Claude's Discretion):** WH_KEYBOARD_LL receives virtual key codes, not characters. VK_W/A/S/D are the same regardless of Shift or CapsLock toggle state — no case sensitivity handling needed.

**Note on CAPSLOCK events:** For CAPSLOCK keydown/keyup events (VK_CAPITAL), continue posting them to the channel with default modifier flags (false, false, false). The modifier flags are only meaningful for direction key events.

```csharp
private static string? GetDirectionName(uint vkCode) => vkCode switch
{
    0x25 => "left",   // VK_LEFT
    0x26 => "up",     // VK_UP
    0x27 => "right",  // VK_RIGHT
    0x28 => "down",   // VK_DOWN
    0x57 => "up",     // VK_W
    0x41 => "left",   // VK_A
    0x53 => "down",   // VK_S
    0x44 => "right",  // VK_D
    _ => null
};
```

Add a `private readonly HashSet<uint> _directionKeysHeld = new();` field to track which direction keys are currently pressed (for key-repeat suppression per user decision).

In `RunAsync`, modify the `await foreach` loop:

```
For each KeyEvent:
1. If evt.VkCode == 0x14 (VK_CAPITAL): handle exactly as today (held/released state machine, onHeld/onReleased callbacks)
2. Else if GetDirectionName(evt.VkCode) returns non-null (it's a direction key):
   a. If evt.IsKeyDown:
      - If _directionKeysHeld already contains evt.VkCode → skip (silent repeat suppression — per user decision "key repeats are suppressed silently, no log spam")
      - Else: add evt.VkCode to _directionKeysHeld
      - If _verbose: log the direction with modifier prefix. Build modifier string from evt.ShiftHeld, evt.CtrlHeld, evt.AltHeld:
        Format: "[HH:mm:ss.fff] Direction: {modifiers}{keyName} -> {directionName}"
        - If no modifiers: "[14:32:01.456] Direction: W -> up"
        - If Shift held: "[14:32:01.456] Direction: Shift+Left -> left"
        - If Ctrl+Alt held: "[14:32:01.456] Direction: Ctrl+Alt+A -> left"
        Per user decision: "Verbose log includes which modifiers were held alongside the direction"
      - Invoke a new optional `Action<string>? _onDirectionKeyDown` callback (constructor parameter), passing the direction name string ("up", "down", "left", "right"). This is the hook point for Phase 8.
   b. If NOT evt.IsKeyDown (keyup):
      - Remove evt.VkCode from _directionKeysHeld (reset repeat tracking)
      - No verbose log for keyup (per user decision: only initial keydown is logged)
      - No callback for keyup
3. Else: ignore (unknown VK code — should not happen but defensive)
```

**Add `_onDirectionKeyDown` callback parameter to constructor:**
